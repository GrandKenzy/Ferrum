# Ferrum Language — Windows AMD64 PE Header Generator
This module implements the PE32+ (64-bit) header generator used by the
Ferrum language toolchain to produce native Windows executables.
The generator creates a complete and valid PE header for the AMD64
architecture. Ferrum uses this header as the base structure when assembling
machine code directly into executable files.
The function only generates the header region. Section contents such as
machine code, static data, import tables, or resources must be generated
and appended separately by the Ferrum assembler or compiler backend.
The output follows the **PE32+** format used by modern 64-bit Windows systems.
---
## Purpose
The generator provides a deterministic PE layout used by the Ferrum
compiler backend. It builds the structures required for Windows to
recognize and load the executable.
The generated header includes:
- DOS Header
- DOS Stub
- PE Signature
- COFF File Header
- Optional Header (PE32+)
- Data Directory Table
- Section Table
Ferrum later appends the compiled machine code and other sections to
produce the final executable.
---
## Main Function
```python
generate(...)
```
Generates the PE header block aligned to `file_alignment`.
The returned value contains **only the header region** of the executable.
Section contents must be written afterwards in the correct order.
**Return value**
```text
bytes
```
A complete PE header block ready to be written at the beginning of
a Windows executable.
---
## Parameters
### `text_size : int`
Virtual size of the `.text` section.
This section contains the executable machine code generated by Ferrum.
The program entry point is automatically placed at the start of this section.
---
### `data_size : int`
Virtual size of the `.data` section.
This section contains initialized global variables.
---
### `bss_size : int`
Virtual size of the `.bss` section.
This section represents uninitialized global memory.
`.bss` does **not** occupy space inside the file because its raw size
is zero.
---
### `other_sections : dict[str, int] | None`
Additional sections to include in the executable.
Example:
```python
{
    ".idata": 512,
    ".rdata": 256
}
```
These sections are appended after `.text`, `.data`, and `.bss`.
---
### `image_base : int`
Preferred base address where the executable will be loaded in memory.
Default:
```text
0x140000000
```
Standard base address used by most Windows AMD64 executables.
---
### `section_alignment : int`
Memory alignment for sections when loaded by the OS.
Default:
```text
0x1000
```
Normally matches the system memory page size.
---
### `file_alignment : int`
Alignment used for section data inside the executable file.
Default:
```text
0x200
```
---
### `subsystem : int`
Specifies which Windows subsystem the executable targets.
Common values:
```text
2 = Windows GUI application
3 = Windows Console application
```
---
### `linker_version : tuple[int, int]`
Version identifier of the Ferrum linker component.
---
### `os_version : tuple[int, int]`
Minimum Windows version required by the executable.
Default:
```text
(6, 0)
```
Corresponds to Windows Vista or newer.
---
### `image_version : tuple[int, int]`
Version identifier of the executable image.
---
### `subsystem_version : tuple[int, int]`
Minimum subsystem version required.
---
### `stack_reserve / stack_commit`
Defines the initial stack memory configuration of the process.
Default values:
```text
reserve = 1 MB
commit  = 4 KB
```
---
### `heap_reserve / heap_commit`
Defines the initial heap memory configuration of the process.
---
### `dll_characteristics : int`
Loader and security flags for the executable.
The default configuration enables:
- NX compatibility
- ASLR support
- Terminal Server awareness
---
### `characteristics : int`
COFF header flags describing properties of the executable.
Default value indicates:
- Executable image
- Large address aware
---
### `timestamp : int`
Timestamp stored in the COFF header.
Ferrum may populate this during builds if reproducibility
is not required.
---
## Section Flags
Standard section flags are defined for common PE sections:
| Section | Purpose |
|-------|-------|
| `.text` | Executable machine code |
| `.data` | Initialized writable data |
| `.bss` | Uninitialized writable data |
| `.rdata` | Read-only data |
| `.idata` | Import table |
| `.edata` | Export table |
| `.rsrc` | Resources |
| `.reloc` | Relocation table |
If a section name is unknown, the generator assigns a default
read-only initialized data flag configuration.
---
## Example Usage
```python
header = generate(
    text_size=len(text_section),
    data_size=len(data_section),
    bss_size=0x1000,
    other_sections={
        ".idata": len(import_table)
    }
)
exe = header + text_section + data_section + import_table
with open("program.exe", "wb") as f:
    f.write(exe)
```
---
## Integration in Ferrum
This generator is used during the final stage of the Ferrum
compilation pipeline:
```text
Ferrum source
     ↓
compiler / assembler
     ↓
machine code generation
     ↓
PE header generation
     ↓
final executable
```
The compiler backend generates section contents and then
combines them with the header produced by this module.
---
## Important Notes
- Section data must be aligned to `file_alignment`.
- Virtual sizes must follow `section_alignment`.
- Sections must be written in the same order used during generation.
- The entry point is placed at the start of the `.text` section.
- Import tables and other structures must be generated separately.
