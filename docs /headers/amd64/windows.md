Ferrum Language — Windows AMD64 PE Header Generator
This module implements the PE32+ (64-bit) header generator used by the
Ferrum language toolchain to produce native Windows executables.
The generator creates a complete and valid PE header for the AMD64
architecture. Ferrum uses this header as the base structure when assembling
machine code directly into executable files.
The function only generates the header region. Section contents such as
machine code, static data, import tables, or resources must be generated
and appended separately by the Ferrum assembler or compiler backend.
The output follows the PE32+ format used by modern 64-bit versions of
Microsoft Windows.
------------------------------------------------------------
Purpose
------------------------------------------------------------
The generator provides a deterministic layout for Windows executables
produced by Ferrum. It handles:
- DOS header and stub
- PE signature
- COFF file header
- Optional header (PE32+)
- Data directories
- Section table
Ferrum's code generation pipeline uses this structure to place machine
code and data into aligned sections while maintaining a valid executable
layout for the Windows loader.
------------------------------------------------------------
Main Function
------------------------------------------------------------
generate(...)
Builds and returns a fully aligned PE header block for a Windows AMD64
executable.
The returned bytes contain the entire header region, including the
section table. After generating the header, the caller must append the
raw section data in the same order used during generation.
Return value:
bytes
    Complete PE header region aligned to file_alignment.
------------------------------------------------------------
Parameters
------------------------------------------------------------
text_size : int
    Virtual size of the .text section.
    This section contains the executable machine code generated by the
    Ferrum compiler or assembler.
    The program entry point is automatically placed at the beginning
    of this section.
data_size : int
    Virtual size of the .data section.
    This section contains initialized global data produced by the
    Ferrum compiler.
bss_size : int
    Virtual size of the .bss section.
    This section represents uninitialized global memory.
    The section does not occupy space in the executable file because
    its raw size is zero.
other_sections : dict[str, int] | None
    Additional sections to include in the executable.
    The dictionary maps section names to their virtual sizes.
    Example:
        {
            ".idata": 512,
            ".rdata": 256
        }
    These sections are appended after the default sections.
image_base : int
    Preferred base address where the executable will be loaded
    in memory.
    Default:
        0x140000000
    This is the typical base address used by 64-bit Windows executables.
section_alignment : int
    Memory alignment used for sections when loaded by the OS.
    Default:
        0x1000
    This normally matches the system memory page size.
file_alignment : int
    Alignment used for sections inside the executable file.
    Default:
        0x200
subsystem : int
    Specifies which Windows subsystem the executable targets.
    Common values:
        2 = Windows GUI application
        3 = Windows console application
linker_version : tuple[int, int]
    Version identifier of the Ferrum linker component.
os_version : tuple[int, int]
    Minimum supported Windows operating system version.
    Default:
        (6, 0)
heap_reserve / heap_commit
    Defines the default heap memory layout used by the process.
stack_reserve / stack_commit
    Defines the initial stack configuration for the process.
dll_characteristics : int
    Security and loader configuration flags.
    The default configuration enables common modern protections
    such as ASLR and NX compatibility.
characteristics : int
    COFF file flags that describe the executable properties.
timestamp : int
    Timestamp stored in the COFF header.
    Ferrum may optionally populate this value during builds.
------------------------------------------------------------
Section Flags
------------------------------------------------------------
The module defines common flags used by the Windows loader for
standard sections:
    .text   executable machine code
    .data   initialized writable data
    .bss    uninitialized writable data
    .rdata  read-only data
    .idata  import table
    .edata  export table
    .rsrc   resources
    .reloc  relocation table
If a section name is not explicitly defined, a default read-only
initialized data flag configuration is applied.
------------------------------------------------------------
Integration in Ferrum
------------------------------------------------------------
Ferrum uses this generator during the final binary assembly stage.
The typical pipeline is:
    Ferrum source
          ↓
    compiler / assembler
          ↓
    machine code + data sections
          ↓
    PE header generation (this module)
          ↓
    final executable layout
After generating the header, Ferrum writes each section block to the
output file using the same ordering defined during header generation.
------------------------------------------------------------
Usage Example
------------------------------------------------------------
header = generate(
    text_size = len(text_section),
    data_size = len(data_section),
    bss_size  = 0x1000,
    other_sections = {
        ".idata": len(import_table)
    }
)
exe = header + text_section + data_section + import_table
with open("program.exe", "wb") as f:
    f.write(exe)
------------------------------------------------------------
Important Requirements
------------------------------------------------------------
1. Section data must be aligned to file_alignment.
2. Virtual sizes must respect section_alignment rules.
3. Sections must be appended in the same order used when generating
   the header.
4. The entry point is automatically set to the beginning of the
   .text section.
5. Import tables, relocations, and other structures must be generated
   by the Ferrum backend before writing the final executable.
